# 竞品动态监控 Agent 产品说明

## 1. 产品定位

竞品动态监控 Agent 是一个面向产品、战略、销售和运营团队的自动化竞品情报工具。当前版本的目标不是替代人工判断，而是把分散在官网、文档、社交媒体、GitHub 等渠道的信息先自动收集、去重、初筛和总结，帮助团队更快发现值得关注的竞品变化。

它适合用于以下场景：

- 每周自动生成指定竞品的动态周报，减少人工巡检多个网站和平台的时间。
- 对高优先级竞品信号进行即时提醒，例如新功能发布、开源项目活跃、生态合作、产品文档变化、社区讨论升温等。
- 把来自不同渠道的零散信息统一整理成可阅读的 Markdown 报告，方便在工作群或内部文档中流转。
- 为产品路线、销售话术、市场观察和战略复盘提供早期线索。

当前版本更偏 MVP 和内部试用阶段，重点验证“多渠道采集 + LLM 初步分析 + 自动报告”的闭环是否成立。

## 2. 当前支持的能力

### 2.1 多竞品监控

系统通过配置文件维护竞品列表和每个竞品的数据源。当前已配置的竞品包括：

- 飞书 Aily
- Coze
- WorkBuddy
- Manus AI
- Notion AI
- Microsoft 365 Copilot
- Skywork
- Claude
- Codex
- Zapier Central
- Make
- n8n

每个竞品可以配置官网、文档、博客、社交平台关键词、GitHub 组织或仓库等数据源。新增竞品主要通过补充配置完成。

### 2.2 多渠道信息采集

当前版本覆盖三类主要信息源：

- 官方渠道：官网、产品页、文档、更新日志、博客、新闻页面。
- 社交和社区渠道：微博、小红书、知乎、B 站、X/Twitter、YouTube、Reddit、Instagram 等，主要通过 TikHub 搜索能力获取。
- GitHub 渠道：支持跟踪组织、单个仓库、Release、Issue、PR、Commit 以及关键词搜索。

这些渠道可以帮助团队同时观察“官方发布”“社区反馈”和“开源生态”三类信号。

### 2.3 自动去重和历史存储

采集到的信息会按 URL 和来源进行去重，并写入本地 SQLite 数据库。这样可以减少重复推送，避免同一条动态在多次运行中反复出现在报告里。

这项能力的实际效果是：

- 周期性运行时只关注新增内容。
- 历史采集结果可以持续沉淀。
- 减少人工处理重复信息的成本。

### 2.4 LLM 结构化分析

每条采集结果会被大模型转换成结构化分析，包括：

- 内容类型：功能发布、博客文章、社区讨论、GitHub 活动、新闻、文档更新等。
- 优先级：high、medium、low。
- 一句话摘要。
- 详细分析。
- 关键信号。
- 潜在影响。
- 建议行动。

这让原始信息从“链接和文本片段”变成更容易被业务团队消费的“竞品信号”。

### 2.5 周报生成

`weekly` 模式会为指定竞品或全部竞品生成 Markdown 周报。周报包含：

- 本周期监测到的新动态数量。
- 高、中、低优先级统计。
- 一句话结论。
- 重点事项。
- 高优先级动态详情。
- 其他值得关注的中低优先级内容。
- 威胁评估。
- 机会评估。

周报适合每周同步给产品、销售、战略和运营团队，用于快速了解竞品近期动向。

### 2.6 实时监控和高优提醒

`monitor` 模式会执行一次实时监控，并筛选出大模型判断为高优先级的动态。高优内容可以通过企业微信、飞书、钉钉机器人推送到工作群。

这项能力适合用于：

- 发现需要当天响应的竞品变化。
- 提醒销售或客户成功团队关注可能影响客户沟通的新信息。
- 将高价值信号从大量低价值社区噪声中筛出来。

### 2.7 通知推送

当前支持的通知方式包括：

- 企业微信机器人。
- 飞书机器人。
- 钉钉机器人。

如果没有配置通知渠道，系统会把结果输出到控制台和日志，便于本地调试。

## 3. 使用方式

### 3.1 前置配置

运行前需要完成以下配置：

- 安装 Python 依赖。
- 配置大模型 API Key、Base URL 和模型名称。
- 如需社交媒体采集，配置 TikHub Token。
- 如需提升 GitHub 访问稳定性，配置 GitHub Token。
- 如需群通知，配置企业微信、飞书或钉钉 Webhook。

### 3.2 生成指定竞品周报

```bash
python main.py weekly Coze
```

运行后会采集 Coze 的全部配置数据源，完成去重、分析和周报生成。若配置了通知机器人，会推送到对应工作群。

### 3.3 生成全部竞品周报

```bash
python main.py weekly
```

系统会按配置逐个处理所有竞品。该模式适合每周定期生成总量较大的竞品监控结果。

### 3.4 执行指定竞品实时监控

```bash
python main.py monitor Coze
```

该模式会执行一次采集和分析，并输出发现的高优先级动态数量。若有高优内容且已配置通知渠道，会自动推送即时简报。

### 3.5 执行全部竞品实时监控

```bash
python main.py monitor
```

系统会逐个处理所有竞品，适合手动触发一次全量巡检。

### 3.6 启动定时任务

```bash
python main.py serve
```

启动后会按配置周期自动运行：

- 默认每周生成周报。
- 默认每 2 小时执行一次实时监控。

具体时间和频率可通过环境变量调整。

## 4. 当前版本能够达到的效果

当前版本已经能够跑通从“信息采集”到“分析产出”的基础闭环，实际价值主要体现在以下方面：

### 4.1 降低日常巡检成本

产品或战略同学不需要每天手动打开多个官网、社交平台和 GitHub 仓库检查动态。系统可以先完成第一轮自动扫描，并把候选信息集中到一个报告里。

### 4.2 提升信息覆盖面

单纯依赖官网更新容易遗漏社区反馈和开源生态变化。当前版本同时覆盖官方、社交、社区和 GitHub 渠道，可以发现更多早期信号。

### 4.3 形成结构化竞品信号

原始网页或帖子往往信息密度不稳定。系统会尝试把每条信息压缩成摘要、影响、建议行动和优先级，降低阅读门槛。

### 4.4 支持团队协作流转

Markdown 周报和群机器人推送天然适合在团队内部传播。相比零散链接，报告更容易被产品、销售、运营和管理层共同阅读。

### 4.5 支持高优内容快速响应

实时监控模式可以把高优先级信号单独推送出来，用于触发后续人工判断，例如更新销售 FAQ、准备竞品对比材料、关注客户迁移风险等。

## 5. 当前局限和已观察到的问题

### 5.1 周报内容不一定严格来自本周

虽然系统会传入近 7 天的时间窗口，但不同平台的 Search API 对时间过滤的支持不一致。有些平台返回的是相关性排序结果，不一定严格按发布时间返回；有些页面无法稳定解析发布时间；还有些接口返回的是“最近抓取到”而非“最近发布”的内容。

因此，当前 weekly 的“本周”更准确地说是“本轮采集到并通过初步时间过滤的新内容”，而不是严格意义上的自然周全量事实集合。

### 5.2 对发布时间的识别还不够强

网页、社交平台和 GitHub 的时间字段格式差异很大。部分网页缺少标准发布时间，系统只能根据页面结构或文本做有限推断。时间缺失时，内容可能被纳入报告但无法明确判断是否属于本周。

### 5.3 Prompt 仍偏基础

当前 Prompt 主要完成单条信息的分类、摘要、优先级判断和建议行动。它已经能提供基础分析，但还不足以稳定产出更深层的战略洞察，例如：

- 不同竞品之间的横向对比。
- 一个竞品连续多周变化趋势。
- 产品能力、商业化、生态、客户群体等维度的体系化拆解。
- 对我方产品路线和销售策略的更具体影响。

### 5.4 报告更像“动态简报”，还不是完整竞品分析

当前 weekly 主要围绕新增条目生成摘要，适合快速阅读。但它还没有形成成熟的竞品分析框架，例如竞争格局、能力矩阵、价格策略、目标客户、渠道动作、生态伙伴和风险等级变化。

### 5.5 社交平台噪声仍然存在

关键词搜索天然会带来噪声，例如无关讨论、广告、转载、旧帖重发、标题党内容等。LLM 可以做一部分降噪，但当前还缺少更稳定的规则层过滤和人工反馈闭环。

### 5.6 采集稳定性依赖外部服务

GitHub、TikHub、官网页面和各类社交平台都可能受到限流、网络环境、反爬策略、接口变更等影响。当前系统能保护单个数据源失败不影响整体流程，但失败源本身仍需要后续增强可观测性和自动恢复能力。

### 5.7 缺少可视化和人工校准

当前主要通过命令行、日志和 Markdown 报告使用。还没有 Web Dashboard，也没有人工标注“这条有用/无用”的反馈入口，长期来看会限制分析质量的持续优化。

## 6. 下一步改进措施

### 6.1 强化时间过滤和新鲜度判断

优先解决 weekly 内容不严格属于本周的问题。建议增加：

- 对每条内容记录 `published_at`、`collected_at`、`source_returned_at` 和 `time_confidence`。
- 报告中明确标注“发布时间未知”“疑似旧内容”“本周新增采集但发布时间较早”。
- 对不同平台建立独立的时间解析策略。
- 对 Search API 返回结果增加二次过滤，不满足时间窗口的内容默认不进入 weekly 主体。
- 周报中区分“本周发布”和“本周发现”。

### 6.2 升级 Prompt 和报告分析框架

将当前单条摘要 Prompt 升级为分层分析：

- 第一层：单条内容事实抽取和优先级判断。
- 第二层：按能力、商业化、客户、生态、技术、市场活动等维度聚类。
- 第三层：生成面向团队的影响判断和建议行动。
- 第四层：和历史周报或其他竞品进行横向比较。

周报可以从“条目汇总”升级为“信号分析”，减少简单罗列。

### 6.3 增加横向竞品对比

新增多竞品对比报告，例如：

- 本周哪些竞品最活跃。
- 哪些竞品集中在 Agent、工作流、MCP、企业协作、开发者工具等方向发力。
- 哪些竞品出现商业化、客户案例或生态合作信号。
- 我方需要重点跟进的 3-5 个竞争动作。

这会让报告从“单个竞品监控”升级为“竞争格局观察”。

### 6.4 建立竞品信号分类体系

建议把动态统一映射到更稳定的业务分类中：

- 产品能力。
- 模型能力。
- 平台生态。
- 开发者工具。
- 企业客户。
- 定价和商业化。
- 渠道和市场活动。
- 开源社区。
- 风险和舆情。

这样周报可以按业务问题组织，而不是只按来源渠道组织。

### 6.5 增加人工反馈闭环

在报告或 Dashboard 中加入人工反馈：

- 有用。
- 无用。
- 误判优先级。
- 时间不准确。
- 需要持续关注。

这些反馈可以用于优化关键词、过滤规则、Prompt 和优先级判断标准。

### 6.6 提升采集质量和源健康监控

建议增加数据源健康状态：

- 最近一次成功采集时间。
- 最近失败原因。
- 连续失败次数。
- 每个 source 的返回条数趋势。
- GitHub rate limit 状态。
- TikHub 或网页采集超时统计。

这样可以判断“没有新内容”到底是真的没有，还是数据源失效。

### 6.7 引入可视化 Dashboard

后续可以提供一个轻量 Web 页面，用于：

- 查看所有竞品近期动态。
- 按竞品、来源、优先级、时间筛选。
- 查看历史周报。
- 标注和反馈结果质量。
- 配置竞品和关键词。
- 查看采集源健康状态。

这会降低非技术同学的使用门槛。

### 6.8 增强通知策略

当前高优内容会直接触发通知，后续可以加入更精细的规则：

- 只推送特定类型的高优动态。
- 对同一主题的多条内容合并推送。
- 对不同团队配置不同通知内容。
- 对低置信度内容先进入待确认队列。
- 增加日报、周报、即时告警三种不同节奏。

### 6.9 沉淀长期趋势分析

随着历史数据积累，可以进一步分析：

- 单个竞品的活跃度变化。
- 高频能力方向。
- 产品发布节奏。
- 社区关注度变化。
- GitHub 活跃度变化。
- 竞品之间的主题重叠和差异。

这会让系统从“信息收集工具”逐步升级为“竞品洞察系统”。

## 7. 当前版本适用边界

当前版本适合作为内部 MVP 使用，帮助团队把竞品监控从纯人工巡检推进到半自动化流程。它可以稳定承担“发现候选信息、做初步摘要、生成周报和提醒”的工作。

但在正式用于高风险决策前，仍建议保留人工复核：

- 关键事实以原文链接为准。
- 高优动态需要人工确认影响范围。
- 周报中的趋势判断应结合历史数据和业务背景。
- 对发布时间未知或来源可信度不足的内容应谨慎使用。

一句话总结：当前版本已经能显著降低竞品信息收集和初筛成本，但要成为真正可靠的竞品分析产品，还需要继续加强时间准确性、分析框架、横向对比、反馈闭环和可视化能力。
